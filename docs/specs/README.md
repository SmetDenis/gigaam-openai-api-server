# Как реализовывать проект по спекам — план и рекомендации

Этот файл — инструкция по исполнению спеков **поэтапно, в отдельных сессиях**.
Источник истины по содержанию — [`00-master.md`](./00-master.md) и этапные спеки `01..07`.

---

## 1. Порядок этапов

```
01 Каркас ──▶ 02 Движок (short) ──▶ 03 Longform (VAD) ──▶ 04 API ──▶ 05 SSE ──▶ 06 Docker/README
                                                                                   └▶ 07 Опт. CPU (опционально)
```
Зависимости жёсткие: каждый этап опирается на предыдущий. 07 — по желанию, после замеров на железе.
**Одна сессия = один этап** (если этап большой — можно разбить по задачам, но не смешивай этапы).

---

## 2. Рабочий цикл одной сессии

1. **Открой новую сессию** в корне репозитория (`gigaam-api`). `CLAUDE.md` подхватится автоматически.
2. **Дай стартовый промпт** (шаблон ниже). Claude прочитает `CLAUDE.md` + `00-master.md` + спек этапа.
3. **План задач:** Claude составляет TodoWrite строго по разделу «Задачи» спека.
4. **TDD по задачам:** тест → реализация → рефакторинг. Идём сверху вниз по списку.
5. **После КАЖДОЙ задачи** — `make pre-commit` (вся пачка тестов всех типов, должно быть зелёным).
6. **Перед завершением этапа** — общий DoD (`00-master.md` §14):
   - зелёный `make pre-commit`;
   - обновлён трекер этапов (`00-master.md` §13);
   - новые архитектурные решения внесены в `CLAUDE.md` (раздел «Архитектурные решения»);
   - `CLAUDE.md` и `README.md` актуальны;
   - удалён мёртвый/«на всякий случай» код.
7. **Коммит — только по явной просьбе.** Claude сам не коммитит.

---

## 3. Шаблон стартового промпта для новой сессии

> Скопируй, подставив номер этапа:

```
Реализуем ЭТАП 0X проекта GigaAM ASR.

1. Прочитай CLAUDE.md, docs/specs/00-master.md и docs/specs/0X-<имя>.md.
2. Составь TodoWrite строго по разделу «Задачи» спека этапа.
3. Иди по задачам в TDD: сначала тест, потом реализация. Тесты — прагматично, без оверинженеринга.
4. После КАЖДОЙ задачи запускай `make pre-commit` и добивайся зелёного.
5. Соблюдай общий Definition of Done (master §14): обнови трекер этапов, внеси новые
   архитектурные решения в CLAUDE.md (ADR-лог), приведи в актуальность CLAUDE.md и README.md,
   удали неиспользуемый код.
6. Не коммить без моей явной просьбы.

Начни с чтения спеков и плана задач, затем покажи план до начала кодинга.
```

## 4. Рекомендации по этапам (подводные камни)

**Этап 01 — Каркас**
- Зафиксируй версии инструментов (ruff/mypy/pytest) в `pyproject.toml`; не плавающие.
- `make pre-commit` на этом этапе гоняет пустые integration-тесты → обработай код выхода 5 как успех.
- Проверь, что `.venv` на Python 3.12 (уже создан), `.python-version` = `3.12`.

**Этап 02 — Движок (короткие аудио)** ⚠️ самый рисковый по зависимостям
- **Ревизия `gigaam`: тегов нет → пин по коммиту `6e4b027`** (проверено 06.2026).
- **onnxruntime (core-dep gigaam, проверено по PyPI):** пин `==1.23.*` имеет колёса `cp312` для
  macOS arm64 (нужен macOS ≥13.0) и Linux x86_64 — ставится и на dev-Mac, и на Synology. **⚠️ Не апгрейдить
  onnxruntime выше пина:** `1.26.0` убрал macOS arm64, `1.24+` убрал macOS x86_64. numpy `==2.*` — ок.
- При сомнении — `uv add` и проба. Если колёса не сходятся — это блокер, решай до движка.
- **Не** подключай `gigaam[longform]` (тянет pyannote). Только torch + сам gigaam.
- На MPS, возможно, нужен `PYTORCH_ENABLE_MPS_FALLBACK=1` — задокументируй; фоллбэк на cpu всегда есть.
- mypy strict вокруг torch: где тип «не выводится», лучше локальный `cast`/обёртка, чем широкий `ignore`.
- Зафиксируй выбранную ревизию gigaam и факт совместимости в ADR-логе CLAUDE.md.

**Этап 03 — Longform (Silero VAD)** ⚠️ самый рисковый по логике
- Алгоритм чанкинга **портируй дословно** из `tmp/GigaAM/gigaam/vad_utils.py::segment_audio_file`
  (сверь `_update_segments`, пороги min/max/strict/new_chunk). Меняем только источник интервалов на Silero.
- `get_speech_timestamps(..., return_seconds=True)` → список `(start,end)` в секундах (как у pyannote).
- **Чанкинг сделай чистой функцией** и покрой юнит-тестами на синтетике — это ядро этапа.
- **Не** вызывай `model.transcribe_longform`. Используй `model.forward` + `model._decode` (master §5.1).
- Память (проверено): int16 целиком (10ч ≈ 1.15 ГБ), float по батчу — для инференса. **Но Silero
  `get_speech_timestamps` берёт весь сигнал во float (≈ 2.3 ГБ/10ч) — пик именно на VAD-стадии**, освобождай
  его до батчей; на длинных у Silero есть рост памяти (issue #356), для очень длинных — `VADIterator` чанками.

**Этап 04 — API**
- Временные файлы загрузок удаляй в `finally` (контекст-менеджер).
- **Upload пиши на диск чанками** (`await file.read(1MB)`), не `await file.read()` целиком — иначе 2 ГБ в RAM
  (проверено: Starlette `UploadFile` грузит весь файл в память). Лимит размера считай по ходу → 413.
- **Очередь/отмена (проверено):** переполнение очереди → 503; ThreadPool-инференс при disconnect клиента
  не прерывается сам — рви longform `is_disconnected()` между батчами.
- `Runner` = один `ThreadPoolExecutor(max_workers=1)` + lock; инференс в executor, loop не блокируется.
- Формат ошибок — строго OpenAI (`{error:{message,type,param,code}}`); коды 400/401/413/415/500.
- Форматтеры времени SRT (`,`) и VTT (`.`) — отдельными функциями, отдельные тесты.

**Этап 05 — SSE**
- Мост «блокирующий генератор сегментов → async» через потокобезопасную очередь; **сериализацию инференса
  не ломай** (по-прежнему 1 за раз).
- `stream=true` + `verbose_json`/`srt`/`vtt` → 400 (стримим только `json`/`text`).
- Проверь: текст из финального `done` == синхронному ответу на тот же файл.

**Этап 06 — Docker/деплой/финальный README**
- **Ключевое:** torch/torchaudio ставить из CPU-индекса (`download.pytorch.org/whl/cpu`), не тянуть CUDA.
- Образ **без весов**; volume `./models:/data/models`; кэш Silero/torch-hub тоже в volume.
- `HEALTHCHECK` на `/health`. `make download-weights` для прогрева.
- **Финальный README** доводится до полного вида (curl/`openai`-примеры, справочник `.env`, деплой, лимиты, troubleshooting).

**Этап 07 — Оптимизация (опционально)**
- Только после реальных замеров. int8 — лишь на cpu; на mps/cuda игнор с предупреждением.
- Зафиксируй результаты в `docs/benchmarks.md` и рекомендацию модели в ADR-логе.

## 5. Общие советы

- **`tmp/GigaAM/`** — справочник по исходникам (model.py, vad_utils.py, decoding.py, __init__.py).
  Это НЕ зависимость: зависимость ставится из git (master §9). В коммиты не попадает (в `.gitignore`).
- **Используй LSP** (go-to-definition/refs/symbols) для навигации по gigaam и своему коду — точнее grep.
- **Версии и API — проверяй фактами:** при сомнениях по torch/fastapi/silero сверяйся через context7
  (офиц. доки) или tavily; не полагайся на память (мир 2026 года меняется).
- **Мелкие задачи:** дроби работу так, чтобы `make pre-commit` гонять часто и ловить регрессии рано.
- **Тесты — прагматично:** покрывай ключевую/рисковую логику и happy-path, без тестов ради тестов.
- **Доки в той же задаче:** меняешь поведение/команды/конфиг → правь CLAUDE.md и README.md сразу.
- **Не золоти:** реализуй ровно требования спека; лишние абстракции/параметры не вводи (кроме `ASREngine`).
- **Коммиты/пуши — только по явной просьбе.**
