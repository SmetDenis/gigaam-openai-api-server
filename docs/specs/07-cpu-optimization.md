# 07 — (Опционально) Оптимизация CPU: int8, бенчмарки, задел под ONNX

> Этапный спек, **опциональный**. Делается после замеров на реальном Synology, если
> производительности не хватает. Общий контекст — [`00-master.md`](./00-master.md), §11–12. Самодостаточен.

## Цель
Ускорить инференс на слабом CPU (Synology, 4 ядра) без потери приемлемого качества:
включить динамическую int8-квантизацию по флагу, измерить RTF/качество на репрезентативных
аудио и подготовить почву под альтернативный ONNX-backend за существующей абстракцией `ASREngine`.
Это этап оптимизации — не блокирует базовый сервис.

## Предусловия
- Завершены этапы 02–06 (рабочий сервис, абстракция `ASREngine`, флаг `QUANTIZE_INT8` уже заведён в конфиге).
- Есть доступ к целевому железу (Synology) или близкому x86_64 CPU для замеров.
- Есть набор тестовых аудио разной длины (короткое, ~минута, длинное) и эталонные транскрипты.

## Артефакты
```
gigaam_api/asr/gigaam_engine.py   # реализация ветки QUANTIZE_INT8 (dynamic quantization)
scripts/benchmark.py              # измерение RTF и (опц.) WER на наборе аудио
docs/benchmarks.md                # результаты замеров (fp32 vs int8; CTC vs RNN-T)
tests/unit/test_quantization.py   # флаг включает квантизацию (на моке/смоук)
# (опционально, отдельной итерацией) gigaam_api/asr/onnx_engine.py — задел под ONNX
```

## Задачи
1. **int8-квантизация** (`gigaam_engine.py`): при `QUANTIZE_INT8=true` на cpu применить
   `torch.ao.quantization.quantize_dynamic` к линейным слоям модели (или к декодеру/энкодеру —
   подобрать безопасный набор слоёв). Только для `DEVICE=cpu`; на mps/cuda — игнор с предупреждением.
   Сохранить корректность типов и совместимость с `forward`/`_decode`.
2. **Бенчмарк** (`scripts/benchmark.py`): прогон набора аудио, измерение RTF (время/длительность),
   пиковой памяти; опционально WER против эталонов (если есть `jiwer`/`editdistance`). Вывод таблицей.
3. **`docs/benchmarks.md`**: зафиксировать результаты: fp32 vs int8, разные модели (CTC/RNN-T),
   рекомендация дефолтной модели для Synology по итогам.
4. **Задел под ONNX** (опционально, может быть вынесено в отдельную сессию):
   - наметить `onnx_engine.py`, реализующий тот же `ASREngine`-протокол через onnxruntime;
   - переиспользовать экспорт GigaAM `model.to_onnx(...)` (см. README GigaAM) и/или `gigaam.onnx_utils`;
   - учесть, что longform/word-timestamps для ONNX потребуют ручной реализации (как в этапе 03) —
     описать объём, не обязательно реализовывать здесь.
   - выбор backend — через конфиг (`ENGINE=pytorch|onnx`), значение по умолчанию `pytorch`.

## Тесты
- **unit** (`test_quantization.py`): при `QUANTIZE_INT8=true` и cpu — путь квантизации вызывается
  (на моке модели); на mps/cuda — предупреждение и пропуск. Смоук: квантизованная модель всё ещё
  отдаёт `ASRResult` на коротком аудио (можно в integration).
- **integration**: сравнить текст fp32 vs int8 на коротком эталоне — деградация в допустимых пределах.

## Debug-логи (этот этап)
- `INFO` — применена ли квантизация, к каким слоям, время; предупреждение при несовместимом device.
- бенчмарк — `INFO` по каждому файлу: длительность, время, RTF, память.

## Acceptance-критерии
- [ ] `QUANTIZE_INT8=true` ускоряет инференс на cpu (зафиксировано в `benchmarks.md`) при приемлемом качестве.
- [ ] Флаг не влияет на корректность типов / `make pre-commit` остаётся зелёным.
- [ ] На mps/cuda флаг безопасно игнорируется с предупреждением.
- [ ] `benchmarks.md` содержит сравнение и рекомендацию модели для Synology.
- [ ] (Если делали ONNX) `onnx_engine` реализует `ASREngine` и переключается конфигом; ограничения задокументированы.
